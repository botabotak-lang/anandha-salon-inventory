# Anandah 在庫・売上・粗利アプリ（MVP）

**更新日**: 2026/03/26

スマホ前提の社内用Webアプリです。数字は **このPC／サーバー内のデータベース（SQLite または PostgreSQL）** に保存されます。ブラウザだけに頼る保存にはしていません。

## 前提

- Node.js 20 以上推奨
- 初期ユーザーはシードで作成（下記）

## セットアップ（ローカル）

```bash
cd Anandha事業/salon-inventory-app
cp .env.example .env   # 既に .env がある場合は不要
npm install
npx prisma db push
npx prisma db seed
npm run dev
```

- 画面: http://localhost:5173  
- API（裏側）: 既定では **ポート 3002**（Vite が `/api` をそこへ中継します）

### 「要ログイン」のまま／すぐログイン画面に戻るとき

だいたい **裏側のAPIが動いていない**か、**ポートの番号がずれている**ときです。

1. **`npm run dev` を一度止めて、もう一度起動**する（画面とAPIが両方立ち上がるコマンドです）。
2. **`.env` ファイルがあるか**確認する。ない場合は `cp .env.example .env` のあと、**`SESSION_SECRET` を32文字以上の好きな英数字**に書き換える。
3. `.env` の **`PORT` は `3002` のまま**にする（`vite.config.ts` の中継先と同じ番号にする）。
4. ブラウザのアドレスは **`http://localhost:5173`** で開く（`127.0.0.1` と `localhost` を行ったり来たりしない）。

ログイン情報は下表のとおりです。

## 在庫の考え方（仮定）

- **物販を売上に載せたとき**: 同じ取引の物販行の数量だけ、自動で **販売出庫** が記録されます。
- **コース付帯の手渡しなど**: 「在庫」画面の **手動出庫**（理由必須）。売上は動きません。
- **取消**: 取引に紐づく販売出庫分は、**入庫で戻す**形で相殺します（履歴は残ります）。

## 物販粗利（ダッシュボード）

- **入金ベース**: 物販の売上明細（税込）から、**その行の数量 × 商品マスタの標準原価** を引いたもの。
- **出庫ベース（参考）**: 当月の **販売出庫＋手動出庫** について、**税込売価（マスタ）×数量 − 原価×数量** の合計。  
  （手渡しのみで売上が付かない月と、入金と出庫のタイミングがずれる月で差が出ます。）

## 手計算チェック例（README用）

1. 商品A: 税込単価 1000円、税率10%、標準原価 400円、在庫 10。  
2. 売上で商品Aを **2個** 登録（税込単価1000のまま）。  
3. **期待**: 物販売上（入金）税込 2000円。入金ベース粗利 2000 − 400×2 = **1200円**。在庫は **8**。  
4. ダッシュボードの「今日／今月」と在庫一覧がこの数と一致すればOK。

## バックアップ（本番）

- **無料ホスト**利用時は、事業者側の自動バックアップの有無・保持期間を必ず確認してください。SLA は一般に限定的です。
- **月1回**: 画面「CSV」から全期間に近い範囲を指定し、**自分のPCやメールに保存**（オーナー合意の二重控え）。
- PostgreSQL 運用時は `pg_dump` の定期実行を推奨（cron・ホストのバックアップ機能など）。

## 本番デプロイの一例

- **SQLite** は同時アクセスに弱いので、本番は **PostgreSQL** 推奨（Railway / Render / Fly.io 等の無料枠＋有料化オプションを README で比較）。  
- フロントは `npm run build` の `dist` を静的ホスティングし、API は別サービス、同一ドメインでクッキーを送る構成が簡単です（サブドメイン間は `SameSite` に注意）。
- スマホで「ホーム画面に追加」しやすいよう、`manifest.webmanifest` と `sw.js` を同梱しています（HTTPS 配信が前提）。

## 2店舗目
